이 스킬은 Docker 컨테이너 내부에서 CPP 또는 Java 앱을 로컬 실행합니다. 장치 배포는 manage-device-app 을 사용하세요.

**자동 라우팅 (v0.6+)**: APP_TYPE 자동 감지 — `<APP>_CPP_SDK/` 가 있으면 **CPP**, `pom.xml` 있으면 **Java**. CPP 는 `app-builder` 이미지에 Platform Service runtime 이 없어 Provider register 가 항상 실패하므로 **자동으로 `--via-fd-cli` 모드로 분기**. Java 는 순수 JVM 이라 그대로 `app-builder` 사용. 강제 우회는 `--use-app-builder` (또는 `USE_APP_BUILDER=1` env).

# Run CPP and Java App Locally

Launch a SeamOS app (CPP or Java codegen-type) inside the app-builder Docker container for local simulation. This is a developer-loop skill — it does not touch physical devices or the SeamOS marketplace.

## Prerequisites

- **Docker daemon** must be running on the host (Docker Desktop on macOS/Windows, Docker Engine on Linux).
- **app-builder image pull access** — the host must be able to `docker pull` the configured app-builder image (online on first run, or pre-loaded via offline bundle).
- **SeamOS project generated by FD Headless** — an existing SeamOS CPP or Java project under `USER_ROOT`, produced by the `create-project` skill (`--codegen-type CPP` or `JAVA`). The skill expects the FD-generated layout (FSP + SDK/APP skeleton). `APP_TYPE` is auto-detected from `FDProject.props` (`JAVA_APP_PATH=` line) or the presence of `<APP>_CPP_SDK/`.
- **python3** (used by `--props` for literal-replace into `Simulator.properties`). Install via Xcode CLT (`xcode-select --install`) or your distro's `python3` package.
- **Rosetta 2 emulation** on Apple Silicon Mac — Docker Desktop → Settings → Features in Development → Use Rosetta for x86_64/amd64 emulation. The app-builder image is `linux/amd64` only; `run-app` pins `--platform linux/amd64` on every `docker run` and emits a best-effort `[run-app] WARN: Apple Silicon detected; Rosetta 2 emulation must be enabled` when the toggle appears off.

## Arguments

| Name | Required | Default | Description |
|------|----------|---------|-------------|
| `APP_NAME` | Required. `--app-name` flag 또는 `APP_NAME` env | — | Name of the SeamOS app project to run (matches the APP project directory produced by FD Headless). CPP/Java 자동 분기. |
| `BIND_ALL` | no | `0` | When `1`, binds the app's listening socket to `0.0.0.0` for host / LAN access. `0` keeps it loopback-only inside the container. |
| `APP_PORT` | no | `6563` | Host-side TCP port exposed for the running app. |
| `WITH_MQTT` | no | `0` | `--with-mqtt` 플래그로 활성화. 로컬 Mosquitto 브로커 컨테이너를 자동 기동해 TestSimulator 가 실제 MQTT publish/subscribe 를 수행하게 한다. |
| `MQTT_DOCKER_IMAGE` | no | `eclipse-mosquitto:2` | `--with-mqtt` 에서 기동할 브로커 이미지. env 로만 override (플래그 없음). |
| `--inject-data <file>` | no | — | host 의 임의 XML 을 컨테이너 안 `com.bosch.fsp.<APP>.gen.tests/data/sample_data.xml` 위에 read-only 로 overlay 한다. host 원본은 절대 수정되지 않으며, 파일 부재 시 exit 2. |
| `--props <key=val>` | no | — | 반복 가능. staging 의 `Simulator.properties` 위에 literal key=value 를 last-win 으로 적용한다. `uiFolderLocation` 같은 강제 키와 충돌 시 stderr 에 `[WARN] --props ${KEY} overrides forced value` 출력 후 사용자 값을 채택. python3 부재 시 exit 3. `--props key` (= 누락) 시 exit 2. |
| `RUNAPP_PLATFORM` | no | `linux/amd64` | docker `--platform` override. 기본값은 Apple Silicon Mac 에서 Rosetta 호환성 유지를 위해 `linux/amd64`. arm64 native 테스트가 필요하면 `RUNAPP_PLATFORM=linux/arm64` 로 export. |
| `RUNAPP_DRYRUN` | no | `0` | `1` 이면 실제 컨테이너/브로커 기동을 생략하고 `docker run …` 명령만 stdout 으로 echo 한다. staging overlay 생성 + cleanup 까지는 실제 수행하므로 회귀 테스트/CI 에서 빠른 sanity 검증에 사용. |
| `USE_APP_BUILDER` | no | `0` | CPP 자동 `--via-fd-cli` 라우팅 비활성화. `--use-app-builder` flag 또는 env `USE_APP_BUILDER=1` 로 활성. 사용처: Java parity 테스트 / app-builder 이미지 자체 디버깅. **CPP 에서 layer 4 실패는 의도된 동작**. |

## Execution

Invoke the wrapper script with flags (또는 env 변수). `--app-name` flag 와 `APP_NAME` env 는 모두 지원됩니다.

> **외부 프로젝트 (USER_ROOT 밖) 실행 시 — 가장 자주 빠지는 함정.**
> `run-app.sh` 는 기본적으로 플러그인 트리(`USER_ROOT`) 안의 프로젝트를 찾습니다.
> 다른 디렉터리(예: `~/work/MyApp/MyApp`)에서 작업 중이라면 매 호출마다
> `APP_PROJECT_ROOT=...` env 를 명시해야 합니다 — 그렇지 않으면 "project not found"
> 또는 잘못된 워크스페이스를 mount 합니다.
> ```bash
> APP_PROJECT_ROOT="$HOME/work/MyApp/MyApp" bash scripts/run-app.sh --app-name MyApp
> APP_PROJECT_ROOT="$HOME/work/MyApp/MyApp" bash scripts/run-app.sh --via-fd-cli --app-name MyApp
> ```
> `--via-fd-cli` 도 동일합니다. 아래 Quick examples 는 플러그인 내부 케이스이므로
> 외부 프로젝트면 모든 예시 앞에 `APP_PROJECT_ROOT=...` 를 붙여서 읽으세요.

Example — env 변수로 `SampleImu2` 실행 (기본 포트, loopback):

```bash
APP_NAME=SampleImu2 bash scripts/run-app.sh --app-name "$APP_NAME"
```

Example — flag 전용으로 모든 옵션 지정:

```bash
bash scripts/run-app.sh --app-name SampleImu2 --bind-all --app-port 6563
```

Example — 로컬 MQTT 브로커와 함께 실행 (`--with-mqtt`):

```bash
bash scripts/run-app.sh --app-name SampleImu2 --with-mqtt
```

Example — 커스텀 sample_data.xml 주입 (`--inject-data`, host 원본 무수정):

```bash
bash scripts/run-app.sh --app-name SampleImu2 --inject-data /tmp/custom_signals.xml
```

Example — Simulator.properties 의 임의 key 를 last-win 으로 override (`--props`):

```bash
bash scripts/run-app.sh --app-name SampleImu2 \
  --props 'logLevel=DEBUG' \
  --props 'http.proxyHost=10.0.0.1'
```

Example — Java codegen-type 앱 실행 (자동 APP_TYPE 분기 + mvn 캐시 host 공유):

```bash
bash scripts/run-app.sh --app-name MyJavaApp
```

Example — arm64 native 컨테이너로 강제 (실험용; 기본은 `linux/amd64`):

```bash
RUNAPP_PLATFORM=linux/arm64 bash scripts/run-app.sh --app-name SampleImu2
```

## fd-cli mode (`--via-fd-cli`)

`app-builder` 이미지에는 NEVONEX Platform Service runtime (`libnevonex-fcal-platform.so`) 이 없어서, 실측 결과 cpp_app 가 `RUNTIME : Requested machine provider:IMUProvider is not yet available` 로 IMUProvider register 에 실패한다 (→ ProcessTimer 미설정 → MainController::run() 침묵 → WS frames 0). 이 한계는 메모리 `project_seamos_provider_injection.md` 에 기록.

`fd-cli` 이미지 (`public.ecr.aws/g0j5z0m9/fd-cli:stable`) 는 같은 Bosch FD 빌드 trunk 의 산출물로, **`/workspace/.nevonex/dependencies/<ver>/lib/`** 에 Platform Service shared lib 가 baked-in 되어 있다. `--via-fd-cli` 는 이 이미지로 build → run → test (TestSimulator) 를 위임해서 `--diagnose` 의 layer 4 까지 PASS 하도록 한다.

> **포트 토폴로지 — 처음 보면 무조건 헷갈리는 부분.** `--via-fd-cli` 모드는 컨테이너에서 **두 개의 서로 다른 서버**가 동시에 뜬다:
>
> | Port | 서버 | 용도 |
> |------|------|------|
> | `:6563` | **Java UI gateway** (Spark/Jetty) | CustomUI 정적 파일 + `get_assigned_ports` 만 서빙. 우리 REST 라우트는 **프록시하지 않음**. |
> | `:1456` | **cpp_app** (NEVONEX) | `registerRoute("/crops", ...)` 로 등록한 모든 REST + `/socket` WebSocket 이 여기에 있음. |
>
> 브라우저 UI 에서 `fetch('/crops')` → `:6563` 으로 가서 404 가 나오는 게 가장 흔한 실수. UI 는 반드시 `get_assigned_ports` 로 **외부 매핑된 1456 포트**를 받은 뒤 `http://${host}:${wsPort}/crops` 로 호출해야 한다 (`seamos-customui-client` 참조). `--diagnose` 는 layer 5 에서만 6563 을 검사하고, layer 3/4 (REST/WS) 는 1456 만 본다.

```bash
bash scripts/run-app.sh --via-fd-cli --app-name SampleImu2
# 다른 터미널 / 같은 셸 후속:
bash scripts/run-app.sh --diagnose

# 외부 프로젝트의 경우 (가장 자주 빠지는 함정):
APP_PROJECT_ROOT="$HOME/work/MyApp/MyApp" \
  bash scripts/run-app.sh --via-fd-cli --app-name MyApp
```

#### `APP_PROJECT_ROOT` 가 가리켜야 할 정확한 디렉토리 (C3)

`<USER_ROOT>/<APP>/<APP>` — FD Headless 가 생성한 **inner `<APP>` 폴더**. 다음 3개 형제가 모두 거기 있어야 한다:

```
$APP_PROJECT_ROOT/                    ← 여기가 APP_PROJECT_ROOT
├── com.bosch.fsp.<APP>/             ← FSP 정의 (FDProject.props 위치)
├── <APP>_CPP_SDK/                   ← 생성된 C++ SDK
└── <APP>_<APP>/                     ← C++ 앱 코드 (CMakeLists.txt, src-gen/, …)
```

자주 발생하는 오류:
- `<USER_ROOT>/<APP>` (한 단계 얕음) — `com.bosch.fsp.*` 형제가 없으니 스크립트가 reject.
- `<USER_ROOT>/<APP>/<APP>/<APP>_<APP>` (한 단계 깊음) — 앱 코드 폴더지 프로젝트 루트가 아님.

| Flag | Default | 설명 |
|------|---------|------|
| `--app-name <NAME>` | required | run-app.sh 와 동일. APP_PROJECT_ROOT 자동 derive. |
| `--image <IMG>` | `public.ecr.aws/g0j5z0m9/fd-cli:stable` | fd-cli 이미지 태그. 환경변수 `FD_CLI_IMAGE` 도 인식. |
| `--skip-build` | off | 이미 빌드된 결과물 재사용 — 빠른 반복 테스트. |
| `--ws-port` | `1456` | host → cpp_app WebSocket. |
| `--ui-port` | `6563` | host → Java UI gateway (Spark Jetty). |
| `--mqtt-port` | `1883` | host → mosquitto. `--with-mqtt` 와 달리 broker 가 host 에 노출됨 → `--diagnose` 가 layer 1, 2 도 검증. |

### 흐름

1. fd-cli 이미지 로컬 부재 시 `docker pull` (linux/amd64).
2. `${APP_PROJECT_ROOT}` 를 컨테이너의 `/workspace/${APP_NAME}` 로 mount.
3. 컨테이너를 detached 로 띄우고 (`sleep infinity` entrypoint), 단계별로 `docker exec`:
   - `fd-commands.sh build ${APP_NAME}` (front-ground)
   - `fd-commands.sh run ${APP_NAME}` (background, log → `/workspace/${APP_NAME}-run.log`)
   - 컨테이너 안 `ss -ltn | grep :1456` 으로 WS readiness 대기 (최대 15s)
   - `fd-commands.sh test ${APP_NAME}` (background, log → `${APP_NAME}-test.log`)
4. 호스트 ports 1456 / 6563 / 1883 publish 완료 → exit 0. 컨테이너는 살아있음.
5. cleanup 은 사용자가 `docker rm -f seamos-fdcli-${APP_NAME_LOWER}`.

### Tradeoffs (`--with-mqtt` vs `--via-fd-cli`)

| 항목 | `--with-mqtt` (app-builder) | `--via-fd-cli` |
|------|--------------------------|---------------|
| Platform Service runtime | 부재 | 베이크됨 |
| 이미지 | 우리가 관리 | Bosch FD trunk 산출물 (변경 시 재 pull) |
| 컨테이너 fork | 앱 + broker 두 컨테이너 | 단일 컨테이너 (broker + 앱 + simulator 한 곳) |
| diagnose layer 4 | FAIL (provider 미register) | **PASS — SampleImu2 로컬 실측 검증** (2026-04-27, mac arm64 Rosetta) |
| FIF / 배포 흐름 적합성 | 낮음 — runtime 다름 | 높음 — 실 디바이스/원격과 같은 Bosch trunk |

## Diagnose mode (`--diagnose`)

이미 실행 중인 앱(Docker `--with-mqtt`, FeatureDesigner host-mode, 또는 임의 환경) 의 데이터 흐름을 5-layer 로 1회 점검한다. APP_NAME 불필요, docker 불필요. broker / WS / UI 포트만 본다.

```bash
bash scripts/run-app.sh --diagnose                         # 모든 기본값 (127.0.0.1)
bash scripts/run-app.sh --diagnose --host 100.110.75.13    # 원격 환경
bash scripts/run-app.sh --diagnose --ui-port 0             # UI gateway 없음 (Docker --with-mqtt 모드 등)
```

| Flag | Default | Description |
|------|---------|-------------|
| `--host` | `127.0.0.1` | 모든 layer 의 target host |
| `--mqtt-port` | `1883` | broker port (layer 1, 2) |
| `--ws-port` | `1456` | cpp_app 의 CustomUI WebSocket port (layer 3, 4) |
| `--ws-path` | `/socket` | WebSocket route (FD 기본값) |
| `--ui-port` | `6563` | TestSimulator Java Jetty UI gateway port (layer 5). `0` 으로 skip. |
| `--topic-filter` | `fek/#` | layer 2 sub 토픽 패턴 |
| `--sample-secs` | `12` | layer 2/4 관측 시간. fd-cli 실측 결과 TestSimulator 의 `fek/3236` publish 간격이 약 10초 — 짧은 window (5–6s) 는 layer 2 false-FAIL 가능. |
| `--skip-broker` | off | layer 1, 2 SKIP. Docker `--with-mqtt` 처럼 broker 가 호스트에 노출되지 않은 경우 사용 — 나머지 layer (3 WS handshake, 4 WS frames, 5 UI HTTP) 만 검증. |

출력 (5 row):

```
[1/5] broker reachable               PASS  127.0.0.1:1883 ($SYS publish counter=57043)
[2/5] topic activity (fek/#)         PASS  10 msgs/5s topics=[fek/3236,fek/3902] first='fek/3236 {"PL":...'
[3/5] WS handshake                   PASS  ws://127.0.0.1:1456/socket → HTTP/1.1 101 Switching Protocols
[4/5] WS frames                      PASS  10 frames/5s sample='{"type":"imu","roll":...}'
[5/5] UI HTTP                        PASS  / → 200 5240B; get_assigned_ports={"1456":1456}
[diagnose] ALL PASS
```

Exit code = 첫 FAIL layer 번호 (1..5), 모두 PASS 면 0. CI/스크립트에서 분기 가능.

**필요 도구**: `mosquitto_sub` (`brew install mosquitto` / `apt install mosquitto-clients`), `curl`, `python3`, GNU `timeout` (macOS 는 `brew install coreutils` 후 `gtimeout` 자동 인식).

**FAIL 해석 가이드**:

| FAIL layer | 원인 후보 | 다음 액션 |
|-----------|----------|----------|
| 1 broker | broker 미기동, port 막힘, ACL 거부 | mosquitto 기동 / 방화벽 / `--mqtt-port` 점검 |
| 2 topic | TestSimulator 미기동 또는 topic prefix 다름 | TestSimulator log / `--topic-filter` 조정 |
| 3 WS handshake | cpp_app 미기동, `/socket` 경로 다름, port 충돌 | cpp_app log / `--ws-path` 점검 |
| 4 WS frames | controller 가 publish 안 함 — interface invalid 또는 ProcessTimer 미설정 (가장 흔한 원인: dev container 의 Platform Service 부재로 Provider register 실패 → `RUNTIME : Requested machine provider:X is not yet available` warning. 자세한 내용은 `memory/project_seamos_provider_injection.md`) | `docker exec <container> tail logs/fcal-cpp-*.log` 에서 위 warning + `Timer already created` 부재 확인 |
| 5 UI HTTP | UI gateway 부재 (Docker `--with-mqtt` 모드는 정상; `--ui-port 0` 으로 skip) | FD 환경이면 Java Jetty 6563 확인 |

## Tradeoff: host m2 캐시 공유

Java codegen-type 실행은 `${HOME}/.m2:/root/.m2` 를 read-write bind 로 마운트해 mvn 의존성 캐시를 host 와 공유한다. 첫 실행에서 의존성을 hydrate 하고 이후 빌드를 incremental 로 가속한다. 단점: 컨테이너에서 host m2 의 임의 artifact 가 보이며, host m2 가 지저분하면 빌드 reproducibility 가 영향을 받는다. 격리가 필요하면 `MAVEN_OPTS="-Dmaven.repo.local=/tmp/m2"` env 로 컨테이너 안에서 별도 디렉터리를 사용하도록 분리할 수 있다 (v0.5.1 후속 설계).
